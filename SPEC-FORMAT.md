# SPEC-FORMAT — 프리셋 스펙 스키마

모든 `presets/<번호>-<코드네임>/SPEC.md`는 아래 12개 섹션을 **전부** 채워야 한다.
비어 있는 섹션이 있으면 그 프리셋은 INDEX.md에서 상태 `spec-draft`를 벗어날 수 없다.

작성 원칙:
- **수치로 말한다.** "여백을 넉넉히" ✕ → "섹션 간 48px, 카드 내부 20px" ○
- **AI가 해석할 여지를 줄인다.** 모든 규칙은 재현 가능해야 한다. 형용사에는 반드시 수치나 예시를 붙인다.
- 코드 예시는 CSS 변수 기준. Tailwind/React는 §11에만.

## 헤더 (frontmatter)

```yaml
---
number: 3                # INDEX.md의 번호와 일치
codename: Jelly
inspired-by: Toss (toss.im) — 서술만, 파일명·클래스명에 상표 사용 금지
mood: 친근한 한국 모바일 핀테크. 크고 대담한 숫자, 넓은 호흡, 말랑한 물성
best-for: 소비자용 모바일 앱, 핀테크, 커머스 결제 플로우
variants: [a, b]         # tokens.css에 존재하는 변형 목록
status: complete | spec-draft | planned
---
```

## §1. 첫인상 요약 (5줄 이내)

이 프리셋으로 만든 화면을 처음 본 사람이 받아야 할 인상.
AI가 세부 규칙 사이에서 길을 잃을 때 돌아올 기준점.

## §2. 색

- **2.1 팔레트**: 시맨틱 토큰 전체 표. `--color-bg`, `--color-surface`, `--color-text`,
  `--color-text-sub`, `--color-primary`, `--color-primary-text`, `--color-accent`,
  `--color-danger`, `--color-success`, `--color-border` (최소 세트, 추가 자유)
- **2.2 라이트/다크**: 양쪽 값 모두. 다크 미지원 프리셋이면 그 이유 명시
- **2.3 대비 규칙**: 본문 4.5:1, 대형/UI 3:1 이상. 프리셋별 추가 제약 (예: "primary 위 텍스트는 항상 흰색")
- **2.4 팔레트 생성 규칙 (변형·랜덤용)**: 새 색 조합을 만들 때 지켜야 할 불변식.
  - 무엇이 고정인가 (예: 뉴트럴 구조, 채도 범위 S 60~80%, 명도 L 45~55%)
  - 무엇이 자유인가 (예: primary hue 0~360 자유, accent는 primary ±30° 이내)
  - 랜덤 요청 처리: 이 규칙 안에서 hue를 고르고, 결과를 `3-r<hue값>` 형태로 명명해 재현 가능하게 한다
    (예: `3-r212` = hue 212 랜덤 변형). 대비 규칙(2.3)을 통과 못 하는 조합은 폐기하고 다시 뽑는다

## §3. 타이포그래피

- 폰트 스택 (한글 포함. 예: Pretendard, system-ui 폴백)
- 타입 스케일 표: 이름 / size / weight / line-height / letter-spacing / 용도
- 위계 규칙: 한 화면에 헤딩 레벨 몇 개까지, 숫자(금액·통계) 전용 처리
- 금지: 임의 font-size 사용 금지 — 스케일에 있는 값만

## §4. 간격 · 레이아웃

- 스페이싱 스케일 (예: 4 8 12 16 20 24 32 40 48 64) — **이 값 외 사용 금지**
- 컨테이너 폭, 그리드 (모바일/데스크톱)
- 화면 기본 패딩, 섹션 간 리듬, 카드 내부 패딩
- 모바일 규칙: 터치 타깃 최소 44px, 하단 고정 CTA 유무와 높이, safe-area 처리

## §5. 형태 (Shape)

- radius 스케일과 적용처 (버튼/카드/입력/모달/썸네일 각각)
- border 사용 여부·두께·색
- shadow/elevation 단계 표 — 정확한 CSS 값으로

## §6. 핵심 컴포넌트 규칙

최소 10종: 버튼(primary/secondary/ghost), 입력 필드, 카드, 리스트 아이템,
상단 네비/앱바, 하단 탭바(모바일), 모달/바텀시트, 토스트, 뱃지/칩, 빈 상태(empty state).
각각: 구조 + 상태(hover/press/disabled/focus) + 치수 + 예시 CSS.

## §7. 모션

- duration/easing 토큰 (예: `--motion-fast: 150ms`, `--ease-out-soft: cubic-bezier(...)`)
- 패턴별 규칙: 진입(entrance), 페이지 전환, hover, press, 리스트 스태거
- 이 프리셋의 모션 성격 한 줄 (예: "스프링감 있는 과장" vs "즉각적이고 절제됨")
- `prefers-reduced-motion` 대응 필수

## §8. 보이스 & 톤 (마이크로카피)

- 어조 정의 + 존댓말/반말, 이모지 사용 여부
- 상황별 예시 표: 버튼 레이블, 에러, 빈 상태, 로딩, 성공 토스트, 확인 다이얼로그
- 금지 표현 (예: "오류가 발생했습니다" 대신 무엇이라 쓰는지)

## §9. 레이아웃 레시피 — 데모 5화면

이 프리셋의 갤러리 데모 5화면 각각에 대해: 화면 목적, 구성 순서(위→아래),
사용하는 컴포넌트, 핵심 치수. 데모 시나리오는 프리셋별 맞춤 (INDEX.md에 기재).
이 레시피는 데모 제작 지시서이자, 실제 앱을 만들 때의 화면 조립 가이드다.

## §10. 금지 규칙 (Anti-AI-look)

이 프리셋에서 절대 하지 말 것. 각 항목은 "AI가 흔히 저지르는 실수 → 대신 할 것" 형태.
공통 예시: 보라→파랑 그라데이션 히어로, 모든 카드에 동일 그림자, 3열 아이콘+제목+설명 반복,
`border: 1px solid #eee` 남발, 이모지 아이콘 대체, 화면마다 다른 radius.
프리셋 고유 금지도 추가 (예: Jelly에서 "촘촘한 데이터 테이블 금지 — 리스트로 풀 것").

## §11. Tailwind v4 / React 매핑 (부록)

- `@theme` 블록으로 tokens.css 변수를 Tailwind 토큰에 매핑하는 코드
- 대표 컴포넌트 2~3개의 React+Tailwind 예시
- StyleSeed 호환 메모: theme.css로 그대로 옮길 때의 대응 관계

## §12. 채점 루브릭

이 프리셋을 따랐다고 주장하는 결과물을 판정하는 기준. loop-runner의 합격 기준으로 그대로 쓸 수 있게 작성한다.

- **verifiable 항목** (기계 판정 가능, 60점 이상 배점): 대비율 계산 통과, 스페이싱 스케일 외 값 0개,
  타입 스케일 외 font-size 0개, radius/shadow가 §5 표의 값과 일치, reduced-motion 대응 존재 등
  — 각 항목에 판정 방법(grep 패턴, 계산식)을 병기한다.
- **subjective 항목** (AI/사람 채점): §1 첫인상과의 부합, §8 어조 준수, §10 금지 규칙의 정성적 항목
  — 각 항목에 1·3·5점 앵커 예시를 쓴다.
- 합격선 명시 (예: verifiable 전항목 통과 + subjective 평균 4.0 이상).

## tokens.css 규약

```css
/* presets/3-jelly/tokens.css */
:root {                      /* 기본 = a 변형, 라이트 */
  --color-primary: #3182f6;
  /* ... §2~§7의 모든 토큰 ... */
}
:root[data-theme="dark"] { /* 다크 오버라이드 */ }
:root[data-variant="b"]  { /* b 변형: 색 계열만 오버라이드 */ }
```

- 변형(a/b/...)은 **색 토큰만** 오버라이드한다. 간격·타이포·모션이 달라지면 그건 변형이 아니라 새 프리셋이다.
- 데모 HTML은 `<html data-preset="3" data-variant="a">`로 시작하고, 토글 JS가 속성만 바꾼다.
