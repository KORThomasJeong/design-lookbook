# WORKFLOWS — 요청 유형별 실행 절차

AI(Claude 등)가 이 저장소를 다룰 때 따르는 플레이북. 요청을 아래 유형에 매핑해 실행한다.

---

## W1. 신규 컨셉 추가 — "OO 사이트 분석해서 N번으로 추가해줘"

1. **번호·코드네임 확정**: `presets/INDEX.md`에서 다음 빈 번호 확인. 코드네임은 상표가 아닌
   자체 이름으로 짓고, 사용자에게 한 번 확인받는다.
2. **사이트 분석**: 대상 사이트(및 공개 디자인 시스템 문서가 있으면 그것 우선)를 조사해
   SPEC-FORMAT.md의 §1~§10을 채울 관찰을 수집한다. 관찰은 반드시 수치화한다
   (스크린샷 눈대중이 아니라: 실제 hex 값, px 값, 폰트 이름, easing 곡선).
   공개 문서가 있는 경우 참조 순서: 공식 디자인 시스템 문서 > 실제 서비스 화면 > 서드파티 분석 글.
3. **SPEC.md 작성**: `presets/<번호>-<코드네임>/SPEC.md`. 12개 섹션 전부(§12 채점 루브릭 포함). 빈 섹션 금지.
4. **tokens.css 작성**: SPEC의 모든 토큰을 CSS 변수로. 라이트/다크 + 기본 변형 a.
5. **INDEX.md 등록**: 상태 `spec-draft`로 추가. 데모 시나리오(5화면 주제)도 이때 정한다.
6. **검증**: 대비 계산(§2.3), 스케일 외 값 없는지 grep, CLAUDE.md 품질 게이트 통과 후
   상태를 유지하거나 W5로 이어 데모까지 만들면 `complete`.

## W2. 색 변형 추가 — "N번에 변형 추가해줘"

1. 해당 SPEC.md §2.4(팔레트 생성 규칙)를 읽는다. 규칙 밖의 색은 만들지 않는다.
2. tokens.css에 `[data-variant="<다음문자>"]` 블록 추가 — **색 토큰만** 오버라이드.
3. 대비 규칙(§2.3) 검증. 실패 시 폐기하고 다시.
4. SPEC.md frontmatter의 `variants`와 INDEX.md 갱신. 갤러리 토글에 변형 버튼 추가(W6).

## W3. 랜덤 팔레트 — "N번 랜덤 색으로 만들어줘"

1. SPEC.md §2.4의 고정/자유 규칙 안에서 hue를 무작위 선택.
2. 결과를 `N-r<hue>`로 명명 (예: `3-r212`) — 같은 이름이면 항상 같은 색이 나오게 재현성 보장.
3. 대비 검증 통과까지 다시 뽑기. 통과한 팔레트를 사용자에게 hex 목록과 함께 보여준다.
4. 일회성이면 저장하지 않고, 사용자가 "이거 고정해줘" 하면 W2 절차로 정식 변형 승격.

## W4. 소비 — "N번(N-x)으로 만들어줘" ★ 가장 흔한 요청 (로컬 스킬 `/lookbook`은 이 절차만 래핑)

1. `presets/INDEX.md`에서 번호 → 프리셋 폴더 해석. 변형 문자 없으면 `a`.
2. 해당 **SPEC.md 전체 + tokens.css를 읽는다.** 요약본이나 기억으로 작업 금지.
   (외부 프로젝트에서 이 저장소가 로컬에 없으면 GitHub raw URL로 fetch — README의 URL 패턴 참조.)
3. tokens.css를 대상 프로젝트에 복사(또는 §11 매핑으로 Tailwind @theme 변환)하고,
   화면은 §9 레시피, 컴포넌트는 §6, 카피는 §8을 따라 생성한다.
4. 완료 전 해당 SPEC.md **§12 채점 루브릭**으로 자체 검수한다. 사용자가 "될 때까지 다듬어줘"를
   원하면 §12를 합격 기준으로 loop-runner(자기개선 루프)에 넘겨 반복한다.
5. 사용자가 "토스풍", "그 핀테크 거"처럼 부르면 INDEX.md의 inspired-by/별칭 열로 매핑한다.

## W5. 데모 화면 제작 — "N번 데모 만들어줘"

1. SPEC.md §9에 정의된 5화면을 `docs/<번호>-<코드네임>/screen-1.html` ~ `screen-5.html`로 제작.
2. 순수 HTML/CSS/JS. 각 화면은 변형·다크 토글 스크립트와 `tokens.css`를 링크한다. 빌드 도구 금지.
3. **`docs/<번호>-<코드네임>/tokens.css`는 `presets/<번호>-<코드네임>/tokens.css` 원본의 복사본이다.**
   토큰 수정은 반드시 presets 원본에서만 하고, 수정 후 docs로 재복사한다(둘은 항상 바이트 동일해야 함).
   docs 복사본을 직접 고치지 않는다 — GitHub Pages가 상대 경로로 서빙하기 위한 복제일 뿐이다.
4. 390px 모바일 우선, 1280px 데스크톱 확인. 모션(§7)과 마이크로카피(§8)까지 구현해야 데모다 —
   정적 스크린샷 수준이면 미완성.
5. 5화면 완성 + §12 채점 루브릭 통과 시 INDEX.md 상태를 `complete`로.

## W6. 갤러리 등록/갱신

1. `docs/index.html` 홈에 프리셋 카드 추가: 번호, 코드네임, 무드 한 줄, 대표 화면 썸네일, 변형 토글.
2. 카드 클릭 → 해당 프리셋의 데모 화면들로 이동. 각 데모 상단에 변형(a/b)·라이트/다크 토글 바.
3. GitHub Pages는 `docs/`를 루트로 서빙한다. 상대 경로만 사용 (커스텀 도메인 가정 금지).

## W7. 스펙 업그레이드 — "OO 사이트 다시 분석해서 N번 업데이트해줘"

1. W1의 분석 절차를 반복하되, 기존 SPEC.md와의 **차이점 목록**을 먼저 만든다.
2. 변경이 색뿐이면 W2(변형)로 처리할지, 스펙 자체를 고칠지 사용자에게 확인한다.
   (기존 번호를 쓰는 프로젝트가 깨질 수 있는 breaking change이기 때문.)
3. 스펙을 고치면 데모(§9 구현물)도 같은 커밋에서 함께 고친다 — 스펙과 갤러리는 항상 동기화.
